# `app_main.c` 阅读说明

`app_main()` 是设备上电后的应用入口。
如果你习惯前端或服务端项目，可以把它理解成“把整个系统装起来的总装配点”。

## 为什么启动顺序这么重要

这里不是普通脚本顺序执行，而是硬件、驱动、UI 和后台任务之间有明显依赖关系：

- NVS 不起来，后面的配置就没法读。
- 板级资源不起来，显示、触摸和共享总线就没法用。
- LVGL 不起来，UI 页面树就没法创建。
- 通信和传感器任务不起来，首页就只会显示占位值。

所以读 `app_main.c` 时，重点不是“调用了什么”，而是“为什么这个顺序不能乱”。

## 主流程怎么理解

### 1. 初始化持久化

这里会先把用户配置、统计信息、默认值等恢复到内存。

### 2. 恢复车型和业务配置

车型会影响：

- 档位推导规则
- 机油温查询策略
- 是否支持某些显示项

### 3. 初始化板级资源

这一段主要是把显示、触摸、共享总线等底座搭起来。

### 4. 初始化 LVGL 运行时

这里会完成：

- LVGL 核心初始化
- 显示驱动接入
- 触摸驱动接入
- 渲染任务和锁的准备

### 5. 创建 UI

这一阶段不是“刷新数据”，而是先把页面、控件树和基础状态准备好。

### 6. 启动通信和采集任务

包括：

- OBD BLE 客户端
- RaceChrono BLE 服务
- 刹车温度采集
- 机油压力采样
- 里程统计等后台任务

到这里系统才算真正“活起来”。

## 阅读时最该盯住的点

- 哪些模块只是初始化一次就结束
- 哪些模块会创建常驻任务
- 哪些全局对象会被后续模块共享
- 哪些初始化失败会让后面的链路一起失效
