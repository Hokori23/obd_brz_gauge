# 第二阶段：产品化与量产准备

## 目标

这一阶段的目标是把“能跑的开发验证样机”推进到“可持续生产的硬件产品方案”。

重点不再是单纯修改代码，而是补齐硬件、电源、结构、生产资料和测试流程。

## 先说结论

这个仓库目前适合做开发验证，不足以直接支持量产。

仓库里当前有：

- 固件源代码
- UI 资源
- 项目文档
- 一套预编译固件

仓库里当前没有看到：

- 原理图
- PCB 工程文件
- Gerber 文件
- BOM
- SMT/装配文件
- 完整接线图
- 结构件图纸
- 量产测试文件

因此，如果要做实际产品，基本需要重新设计硬件，并补齐一整套生产资料。

## 第二阶段需要哪些硬件

### 1. 主控板

量产阶段不建议继续直接使用开发板，而是建议：

- 继续使用 `ESP32-S3` 作为主控
- 选择 `ESP32-S3` 模组上自研主板

这样做的原因：

- 软件迁移成本低
- BLE、UI 和当前工程可继续复用
- 可控性比现成开发板高

### 2. 显示与触摸模组

你需要重新选择真正能稳定供货的显示方案：

- 屏幕尺寸
- 分辨率
- 驱动 IC
- 触摸芯片
- 亮度
- 安装方式
- 供应链稳定性

如果未来量产屏幕与当前仓库目标硬件不同，需重新做驱动适配与 UI 验证。

### 3. 电源模块

车载场景不能直接按 USB 5V 的开发习惯处理，必须补齐：

- `12V` 输入降压
- 反接保护
- 过压保护
- 浪涌与瞬态保护
- 上下电行为控制
- 地线与噪声处理

这是产品化阶段必须补的部分，否则上车稳定性会很差。

### 3.1 车载可靠性与合规前提

如果目标不是个人样机，而是长期装车、可卖或可批量交付的设备，还需要把这些当成必选项：

- `ESD` 防护
- `EMC/EMI` 设计与整改
- `load dump` 与冷启动压降等车载电源异常场景
- 工作温度与存储温度验证
- 振动、跌落、连接器寿命与线束可靠性
- 阻燃、绝缘与基本安全设计
- 目标市场需要满足的法规与认证要求

也就是说，“能打板”和“接近可量产”之间还有一大段系统工程工作，不应混为一谈。

### 4. OBD 数据接入方案

你需要先决定产品路线：

- 方案 A：继续依赖外置 `BLE ELM327`
- 方案 B：主板直接接入车载总线，自己做 OBD/CAN 接口

注意：

- 方案 A 是当前仓库软件架构可以直接延续的路线
- 方案 B 不是当前仓库的小改，而是新的硬件和固件项目
- 如果走方案 B，通常需要新增或重做 `CAN` 收发、电源保护、协议栈、诊断会话、数据解析与整车兼容性验证
- 当前仓库并没有现成的 `CAN/ISO-TP/UDS` 直连接入实现

通常：

- 想快速复刻，先走方案 A
- 想更像正式产品，长期更可能走方案 B

方案 A 的优点是开发快，缺点是依赖外置盒子，用户体验和稳定性受第三方设备影响。

### 5. 扩展传感器接口

如果产品要保留扩展能力，需要明确：

- 是否保留 `RS485` 刹车温度
- 是否保留油压采集
- 是否支持更多模拟/数字传感器
- 接口形式是端子、插头还是线束

## 是否需要打板

需要。

如果你想从验证样机走到实际产品，至少应准备三类板：

### 第 1 版：功能验证板

目标：

- 集成 `ESP32-S3`
- 集成屏幕与触摸接口
- 集成调试口
- 集成基础电源
- 预留 OBD 与传感器接口

这版主要验证电路方向是否正确，不追求外观与结构成熟。

### 第 2 版：工程样机板

目标：

- 调整尺寸与接口布局
- 优化供电与稳定性
- 适配结构件
- 减少飞线和临时转接

这版开始逼近最终产品形态。

### 第 3 版：小批量试产板

目标：

- 修复前两版问题
- 验证装配、烧录、测试流程
- 验证供应链稳定性

## 第二阶段需要补齐的资料

如果要交给板厂、装配厂或硬件工程师，至少要准备：

- 系统硬件框图
- 原理图
- PCB
- Gerber
- BOM
- 关键器件选型表
- 连接器与线束定义
- 外壳与结构设计图
- 烧录与测试流程
- 生产测试治具方案

## 这个仓库里有没有打板和组装信息

目前没有看到成套可生产资料。

现有仓库只能提供这些参考价值：

- 当前软件基于什么主控和显示路径
- 当前 UI 长什么样
- 当前 OBD/BLE 逻辑如何组织
- RS485、油压等扩展功能需要哪些软件模块

但它不能直接回答这些生产问题：

- 板子怎么画
- 器件怎么选
- 电源怎么做
- 屏幕怎么固定
- 整机怎么装配
- 批量测试怎么做

## 推荐的实际推进顺序

### 第 1 步：先完成第一阶段

不要在开发验证还没跑通时就开始打板，否则大概率会把软件问题和硬件问题混在一起。

### 第 2 步：定义最小产品范围

先明确第一版产品必须有什么，不需要有什么。

建议至少先决定：

- 是否只支持 `BRZ ZC6`
- 是否只显示基础 OBD 数据
- 是否保留外置 `ELM327`
- 是否保留 RS485 和油压扩展

### 第 3 步：输出硬件框图

在画原理图前，先把模块边界画清楚：

- 主控
- 电源
- 显示
- 触摸
- OBD
- 传感器接口
- 调试与升级接口

### 第 4 步：找硬件工程资源

如果你没有单片机硬件经验，建议尽早引入能画板的硬件工程师。

软件可以先自己继续熟悉，但量产板、电源和车载可靠性最好不要完全靠试错。

## 第二阶段的交付物建议

当你推进到量产准备时，建议把目标收敛到这些交付物：

- 一份产品功能定义文档
- 一份系统框图
- 一版可打样的原理图和 PCB
- 一份初版 BOM
- 一份装配和接线说明
- 一份测试流程说明

## 风险提醒

产品化阶段最大的风险通常不是“代码写不出来”，而是：

- 车载供电不稳定
- 连接器与线束不可靠
- 屏幕和触摸供应不稳定
- 外置 OBD 盒子兼容性差
- 环境温度、振动和长期使用导致异常

所以第二阶段的重点应该是“系统工程”，而不只是“把当前仓库代码继续改下去”。
